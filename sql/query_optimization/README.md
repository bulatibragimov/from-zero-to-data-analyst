# SQL Query Optimization

## Overview

В этом разделе рассматриваются основные методы оптимизации SQL-запросов для повышения производительности при работе с большими таблицами.

Оптимизация запросов является важной частью работы аналитика данных, поскольку неправильно написанный запрос может значительно увеличить нагрузку на базу данных и замедлить выполнение аналитических задач.

Основная цель оптимизации — уменьшить объём обрабатываемых данных и выбрать наиболее эффективный способ выполнения запроса.

---

# Key SQL Concepts

## LIMIT

`LIMIT` используется для ограничения количества возвращаемых строк.

Во время разработки запросов рекомендуется всегда использовать `LIMIT`, чтобы:

* избежать случайной обработки миллионов строк
* быстро проверить корректность логики запроса
* ускорить тестирование запросов

Пример:

```sql
SELECT *
FROM orders
LIMIT 100;
```

---

## EXPLAIN

`EXPLAIN` показывает план выполнения запроса без его запуска.

Это позволяет понять:

* какие индексы используются
* какие типы сканирования применяются
* в каком порядке соединяются таблицы

Пример:

```sql
EXPLAIN
SELECT *
FROM orders
WHERE customer_id = 100;
```

---

## EXPLAIN ANALYZE

`EXPLAIN ANALYZE` выполняет запрос и показывает реальное время выполнения каждой операции.

С его помощью можно:

* выявить медленные части запроса
* сравнить ожидаемое и фактическое количество строк
* обнаружить неэффективные операции

---

## Filtering Early

Главное правило оптимизации SQL:

**Фильтровать данные как можно раньше.**

Условия `WHERE` должны максимально уменьшать количество строк перед выполнением:

* `JOIN`
* `GROUP BY`
* `ORDER BY`

Это снижает нагрузку на базу данных.

---

## JOIN Optimization

`JOIN` — одна из самых ресурсоёмких операций.

Для оптимизации соединений рекомендуется:

* фильтровать данные в каждой таблице до выполнения `JOIN`
* использовать индексы на колонках соединения
* соединять уже уменьшенные наборы данных

---

## CTE vs Subqueries

В большинстве случаев оптимизатор базы данных обрабатывает `CTE` и подзапросы одинаково.

CTE используются в основном для:

* повышения читаемости кода
* структурирования сложных запросов
* повторного использования логики

---

## Indexes

Индекс — это структура данных, которая ускоряет поиск строк в таблице.

Индексы особенно полезны для:

* `WHERE`
* `JOIN`
* `ORDER BY`

Однако индексы также замедляют операции `INSERT` и `UPDATE`, поэтому их следует создавать только для часто используемых колонок.

---

## Partitioning

Партиционирование — это разделение большой таблицы на несколько меньших частей.

Обычно используется для:

* таблиц с большим объёмом данных
* данных, которые можно разделить по дате
* ускорения аналитических запросов

---

## Distributed Databases

В распределённых системах данные хранятся на нескольких серверах.

Это позволяет:

* масштабировать систему
* выполнять запросы параллельно
* обрабатывать огромные объёмы данных

---

## Best Practices

При написании SQL-кода рекомендуется:

* писать ключевые слова в верхнем регистре
* использовать понятные алиасы таблиц
* делать переносы строк
* добавлять комментарии к сложной логике
* избегать `SELECT *`
* проверять план выполнения запроса

---

## Query Development Checklist

Перед запуском SQL-запроса стоит проверить:

1. Нужны ли все выбранные столбцы.
2. Используются ли индексы в `WHERE`.
3. Есть ли `LIMIT` для тестирования.
4. Понятно ли оформлен код.
5. Проверен ли план выполнения через `EXPLAIN ANALYZE`.

Оптимизация SQL-запросов является важным навыком для аналитиков данных и позволяет эффективно работать с большими объёмами информации.
